VMD-AI · Research Progress Report

Completion is not correctness: verifying an LLM agent that drives molecular visualization

A self-hosted local model, an oracle-backed correctness scorer, and a 2×2 RAG × Wiki ablation on VMD analysis tasks.

Author: Peter  ·  System: vmd_ai (ChatVMD) agent  ·  Date: June 12, 2026  ·  Structure: 1CRN (crambin, 46 res / 327 atoms)

Executive summary

  1. Completion ≠ correctness. An outcome-style benchmark (SciVisAgentBench) scores the agent ~100% by checking that an answer was reported; oracle verification shows a large share of those analysis answers are physically wrong.
  2. Correctness does not track model tier. A free local Qwen-72B computed the radius of gyration exactly (9.666 Å) where a frontier model earlier returned 329 Å; on contact-counting the order reverses. The text rubric passes both.
  3. Zero-cost evaluation works. Qwen2.5-72B (4-bit) self-hosted on a single A100 runs the full benchmark with no per-token cost, ~9 s/case once GPU contention is removed.
  4. Retrieval doesn't help — but injected knowledge does. The agent never calls the docs_search tool, even when explicitly ordered to, so RAG ≈ baseline. Placing the same reference directly in context lifts correctness 3.6 → 4.4 / 5 and cuts its variance threefold. The lesson is to enforce knowledge, not offer it.
  5. And it generalizes. Across five proteins (327–9,058 atoms), injected knowledge raises correctness 42 → 55 / 75, with the gain concentrated on the syntax-hard measures (CA–CA distance, SASA) — the same pattern seen on 1CRN.
  6. The gap is on both halves. Visualization is checked the same way (scene-state assertions, not pixels): the baseline agent wrote "yes, licorice" while building zero representations — a false "yes" the rubric passes and scene-state catches. So the agent will claim a picture it never drew, not just a number it never computed.

1 · Background

vmd_ai is an LLM agent that controls VMD (Visual Molecular Dynamics) through an agentic tool-calling loop: the model issues Tcl via a run_vmd_command tool, inspects the scene, and saves images and answers. The open question for the project was how to evaluate such an agent. Existing agent benchmarks score task completion — typically via an LLM judge that checks whether a plausible answer was produced. This report argues that for scientific tasks, completion is the wrong target, and demonstrates an oracle-backed alternative that measures whether the agent's numbers are actually right.

2 · Setup

  • Agent: vmd_ai / ChatVMD tool loop, run unmodified, driving a headless native VMD 1.9.4 through a pseudo-terminal bridge.
  • Model: Qwen/Qwen2.5-72B-Instruct-AWQ (4-bit), self-hosted with vLLM on one NVIDIA A100-80GB; OpenAI-compatible API, Hermes tool-call parser. No API cost.
  • Tasks: the molecular_vis suite from SciVisAgentBench (10 cases on 1CRN): six visualization cases (cases 1–6) and four quantitative analysis cases (RMSD/RMSF, radius of gyration, distance + dihedrals, contacts).
  • Ground truth: a VMD oracle (gold_oracle.tcl) recomputes the true values every run — never hand-typed.

Oracle gold for 1CRN: Rg = 9.6663 Å · RMSD(self) = 0 · CA(1)–CA(10) = 12.603 Å · φ/ψ(res 5) = −76.18° / −18.98° · contacts(≤8 Å, all-atom) = 10 835.

3 · Finding 1 — completion is not correctness

The benchmark's molecular_vis text rubric asks, in effect, “did the output report a numeric value for the calculated quantity?” — so a wrong number scores full marks. Two demonstrations from this study:

Every one of these passes a completion check. Only an oracle separates the exact Rg from the contact count that is wrong by three orders of magnitude. This is the core argument for programmatic verification (the approach in the companion vmdbench) over outcome LLM-judging.

5 · Finding 3 — retrieval doesn't help this agent; injected knowledge does

A 2×2 factorial — {document RAG} × {persistent Wiki} — was scored for correctness against the oracle. An early 3-seed pass suggested RAG added +1.7 / 5; five-seed re-runs overturned that — the gain was an under-powered artifact — and prompted a three-arm retrieval study that produced a sharper, more useful result.

Corrected by more seeds. At 3 seeds RAG looked like 2.0 → 3.7 / 5. At 5 seeds the RAG arm sits at or below baseline (e.g. none 2.8 ±1.2 vs rag 2.2 ±0.4). A cautionary note on small-sample agent benchmarks: the headline reversed when the seed count doubled.

So why doesn't retrieval help? Four arms, each against the same baseline, isolate it — and the deciding variable is when the knowledge arrives:

RAG armknowledge deliverycase_9correctness /5
ragoptional tool — model may call docs_search✗≈ none
ragforcetool + prompt orders "retrieve first"✗≈ none
autoragretrieved after an error (reactive)✗3.0 ±1.3
smartinjectproactive, but only the retrieved subset✗2.4 ±1.4

Two factors decide it. Timing: proactive inject beats reactive autorag beats the never-invoked tools. Completeness: smartinject retrieves only the top-2 sections, dropping idioms a multi-step task also needs (the φ/ψ recipe, file mkdir) — so injecting the whole small cheatsheet wins. Subset-retrieval only pays off on a large corpus; a curated cheatsheet should be injected entire.

The index retrieves perfectly — a query for "measure dihedral" returns the exact syntax chunk at the top (BM25 score 13.2) — so the failure is neither retrieval quality nor the model's ability to use docs. The agent simply never invokes the tool: not on its own, and not when explicitly instructed. But when the same reference is forced into context, correctness climbs and its variance collapses, fixing the hard cases the tool-based arms could not:

The lesson: enforce knowledge, don't offer it inject 3.6 → 4.4

For a weaker local model an optional retrieval tool is dead weight — it won't call it even when ordered to. What matters is timing: the same reference fixes case_9 when injected before the model acts (proactive, 4.4/5), but not when surfaced after it errors (reactive autorag, 3.0/5) — by then it is mid-failure and this model doesn't recover. And for a small curated corpus, completeness beats precision — retrieving only the relevant subset (smartinject, 2.4/5) drops idioms a multi-step task also needs, so the whole cheatsheet wins. This mirrors DocPrompting (ICLR 2023), which conditions generation on retrieved docs up front rather than leaving the choice to the model. The fix is proactive injection of the full curated reference (or fine-tuning), not a retrieval tool or a cleverer index.

The Wiki hurts — from a cold start −1.0 / 5 (3 seeds)

The self-maintained Wiki was the most reliable-looking arm (10/10 completed) yet the least correct (1.0 / 5), even breaking cases the model otherwise nails — its ~3× tool-call overhead crowds out the task. But the ablation reset the Wiki to empty every run, so this measures cold-start cost, not the cross-session accumulation it is designed for. The warm-Wiki case remains untested.

6 · Generalization — the win holds beyond 1CRN

To check the findings aren't a single-structure artifact, the five portable metrics (Rg, atom/residue counts, first↔last CA distance, SASA) were run over five proteins spanning 327–9,058 protein atoms (1CRN, 1UBQ, 1HSG, 1HCK, 1QMZ), 3 seeds per task, none vs inject, scored against a structure-parametric oracle.

Generalizes — and concentrated where the Tcl is hard +13 / 75

Injection helps across all five structures, and the gain lands exactly where the 1CRN study predicted: the syntax-error-prone measures — CA–CA distance (+8) and SASA (+5), both bracket-heavy measure calls — while the single-command easy metrics (Rg, atom count) are flat within noise. The same lesson, now on proteins up to 9,058 atoms: the cheatsheet earns its keep on the hard idioms, not the trivial ones.

Server-noise caveat. Both arms carry transient "no answer" failures (timeouts/thrash on a contended shared GPU — visible even on tiny 1CRN), which drag both totals down; whenever the agent does answer it is exact to the decimal. So absolute scores understate true correctness, but the inject-vs-none gap and the per-metric concentration are the robust signal. A dedicated GPU and more seeds would lift both arms and sharpen the margin.

7 · Visualization correctness — the gap is on both halves

Everything above scores analysis (numbers vs an oracle). Visualization is checkable the same way — not by pixels or a vision judge, but by scene-state assertions: replay the agent's Tcl and verify the representation it actually built (style / coloring / selection). It's the visual analog of scalar_within — deterministic and implementation-agnostic.

A false "yes," caught baseline: licorice built nothing

On the Licorice task the baseline agent wrote "yes" — the text rubric passes — but scene-state shows it built zero representations (the recurring mol load cif load failure, so nothing rendered). Injection fixes it (a valid mol new load → the rep exists), 4/4. So the completion-vs-correctness gap is not only in the numbers: the agent will claim a picture it never drew, and only scene-state verification catches it.

Aside: visualization is the agent's stronger half — representation Tcl (mol modstyle/modcolor/modselect) is flat and simple, whereas its analysis failures were all bracket-heavy measure calls. Single pass; the discriminating result is the Licorice false-yes.

8 · Making the agent correct — and what actually generalizes

Three targeted fixes, from the failure analysis: (1) inject a curated VMD-Tcl cheatsheet up front; (2) reject syntactically incomplete Tcl before it hangs the interpreter, and raise timeouts; (3) expose semantic tools (vmd_measure, vmd_represent) that emit the correct Tcl so the model only names the operation. Together they take Qwen from ~56% to 97% correctness with the completion–correctness gap collapsing to 0 on the five-structure analysis suite — when it answers, it is right.

Rigor — is that generalization or tool coverage? The semantic tools' operations are exactly the suite's metrics, so 97% partly reflects "the tool does the task." We re-evaluated on held-out operations the tools do not cover (hydrogen bonds, a 3-atom angle, a dihedral, centre-of-mass, bounding-box extent, a GLY count), decomposing none / inject-only / +tools.

What generalizes, precisely

The 97% is largely tool coverage: strip the covering tools and correctness falls to 67% (24/36). The semantic tools add nothing off their coverage (inject-only = +tools, both 24/36). The real generalization is from injection (21→24), and only on the idioms the cheatsheet teaches — the 3-atom angle (4→6) and dihedral (3→6). Operations it never mentions are unaffected (hbonds, centre-of-mass: solved regardless) or resist everything (GLY count: 0/6). And injection is not free: it slightly hurt the bounding-box task (2→0) — an injected reference can distract on operations it doesn't cover.

Bottom line: semantic tools are a strong fix when they cover the operation; injection gives a smaller, idiom-specific generalization; some operations stay hard regardless. The headline 97% is the former, not a general capability jump.

9 · Methodology & reproducibility

  • Oracle gold is regenerated by VMD on every scoring run; agent answers are parsed from their saved files and compared within per-metric tolerances.
  • Gold is cross-validated against two independent tools. MDAnalysis and Biopython (both unrelated to VMD) recompute Rg, atom/residue counts and CA–CA distance across all five structures (Biopython reads the 1CRN mmCIF). 19 of 20 checks agree — atom counts, residue counts and CA–CA distances exactly, Rg to <0.07 Å (within mass-model drift). The single difference is the 1QMZ residue count (VMD 1124 vs 1122): VMD's protein macro includes 2 modified residues the other tools exclude — a definition choice, not an error.
  • Seeds. Up to five repeats per arm; the per-case pass-rate exposes flaky cases (e.g. case 9). This mattered directly: a 3-seed run suggested RAG added +1.7, which five seeds overturned — small samples mislead on stochastic agents.
  • Result integrity. Result JSONs are timestamped and append-only; every (arm, seed) and every sweep is namespaced, so no run overwrites another. Each sweep's scoreboard is archived.
  • Contacts (case 10) is reported but excluded from the scored total: the “native contacts” definition is sensitive and must be pinned to the task's intended CA–CA form.

9 · Caveats & next steps

  • Statistical power: 5 seeds, one structure (1CRN); spreads remain non-trivial. The injection win (3.6 → 4.4) is the most robust effect — broaden to more structures and 7–10 seeds to firm it up.
  • Enforce, don't request: the agent ignores an optional docs_search tool even when explicitly ordered, yet uses the same knowledge correctly when it is injected. The productive direction is to force the knowledge in (always-inject, tool-choice forcing, or fine-tune on the correct idioms), not to improve the index.
  • Why the Wiki hurts needs a transcript-level look (turn budget spent on wiki_update vs answering?), plus a warm-start run that lets it accumulate across sessions — the case the reset hid.
  • Clean cross-model baseline: rerun Claude through the same correctness scorer to lock Finding 2.
